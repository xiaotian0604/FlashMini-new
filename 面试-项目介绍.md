# Flashmini 项目介绍（面试口述版）

> 以第一人称口语化表达，适合面试中 3-5 分钟项目介绍使用

---

## 一、项目背景——为什么要做这个东西

我们团队同时维护着四五条小程序业务线，涵盖微信、支付宝、百度、字节四个平台。之前每次发版，开发者要手动打开每个平台的开发者工具，一个一个上传，填版本号、写备注，然后再去飞书群里手动通知测试同学。一次完整的四平台发布下来，大概要花 15 到 20 分钟，而且过程中很容易出错——比如版本号忘了改、上传了错误的构建产物、某个平台漏传了。

我就在想，这个事情完全可以自动化。市面上有一些零散的单平台 CI 工具，比如微信的 miniprogram-ci，但没有一个统一的多平台解决方案。于是我决定从零搭建一个 CLI 工具，把多平台上传、配置管理、通知推送这些事情一站式解决。

---

## 二、整体设计思路——我是怎么规划的

动手之前我先想清楚了几个核心原则：

**第一，轻量。** 这是一个工具类项目，不是业务项目，不能引入太多依赖。我给自己定了一个目标——核心包体积控制在 500KB 以内。所以 HTTP 请求直接用 Node 18 的原生 fetch，不引 axios；CLI 框架用 Commander.js 而不是更重的 Yargs。

**第二，可扩展。** 不同业务线的需求不一样，有的团队上传前要跑 lint 检查，有的上传后要写日志文件，有的要推企业微信而不是飞书。这些差异化需求不能硬编码在核心流程里，所以我设计了一套插件系统，让用户可以在任意生命周期阶段注入自定义逻辑。

**第三，类型安全。** 配置文件是用户手写的，很容易写错。以前我们遇到过好几次因为 appId 写错了导致上传失败、半小时白忙活的情况。所以我用 Zod 做了运行时配置校验，错误信息精确到字段级别。比如微信的 appId 必须以 wx 开头，你写错了它会直接告诉你"微信 AppID 必须以 wx 开头"，而不是等到调微信 SDK 的时候报一个看不懂的错。

**第四，CI 友好。** 工具最终是要嵌到 CI 流水线里的，所以我做了非零退出码、JSON 输出模式、环境变量注入这些 CI 标配能力。

---

## 三、架构设计——四层架构怎么分的

我把整个项目分成了四层，自上而下依赖方向单向：

**最上面是 CLI / API 入口层。** 这一层负责两件事：一是用 Commander.js 解析命令行参数，对外暴露 `flashmini upload`、`flashmini init`、`flashmini preview` 三个命令；二是导出 Node.js API，让用户可以在代码里 `import { upload } from 'flashmini'` 直接调用。这就是所谓的"双调用模式"——CLI 和 API 共享同一套核心逻辑，只是入口不同。

**第二层是核心调度层。** 这是整个项目的大脑。它的核心是一个 Runner 类，负责编排完整的上传流程：加载配置 → Zod 校验 → 加载插件 → 解析版本号和 git 信息 → 执行 beforeAll 钩子 → 分发到各平台上传 → 执行 afterAll 钩子 → 发送通知。Runner 本身不包含任何平台特定的逻辑，它只做流程编排。

**第三层是插件系统和平台策略层，这两个是并列的。** 插件系统提供生命周期钩子，平台策略层封装各平台的上传逻辑。它们都向核心调度层提供能力，但彼此不直接依赖。

**最底层是通知系统。** 上传完成后，通知系统读取上下文中的结果数据，构建消息并发送到飞书、钉钉或自定义 Webhook。

这种分层的好处是职责清晰，每一层只关心自己的事情。比如我要新增一个快手小程序平台，只需要在平台策略层新建一个文件，核心调度层完全不用动。

---

## 四、贯穿全流程的上下文对象

这里有一个设计我觉得比较重要，就是 Context 对象。它是一个贯穿整个上传生命周期的数据载体。

为什么要有这个东西？因为整个流程中，不同模块需要共享很多状态——版本号、git 分支、上传结果、插件写入的自定义数据。如果用全局变量，那代码耦合度太高，测试也很难写。所以我设计了一个 Context 类，在 Runner.run() 的时候创建，然后一路传递下去。

Context 上有 config（只读的配置）、version、description、gitBranch、results 这些字段，还有一个 extra 字段供插件存放自定义数据。它还提供了一些计算属性，比如 successCount、failedPlatforms、duration，方便其他模块直接使用。

这其实就是上下文对象模式，好处是数据流向非常清晰——谁读了什么、谁写了什么，一目了然。

---

## 五、插件系统——我认为最有设计感的部分

插件系统是我花心思最多的模块。它参考了 Webpack 的 Tapable 设计理念，但我没有直接用 Tapable，因为 Tapable 是 webpack 的内部包，引进来会带一堆不需要的依赖。所以我自己实现了一个轻量版，核心代码不到一百行。

**核心是 HookManager 类**，它内部用一个 Map 存储钩子名到回调函数数组的映射。提供三个核心方法：

- `tap(hookName, fn)` —— 插件用它注册回调
- `call(hookName, ...args)` —— Runner 用它串行执行所有回调
- `waterfall(hookName, initial, ...args)` —— 瀑布流执行，每个回调的返回值传给下一个，用于 beforeNotify 钩子让插件能链式修改通知消息内容

我定义了 6 个生命周期钩子：beforeAll、beforeUpload、afterUpload、afterAll、onError、beforeNotify。它们覆盖了上传流程的所有关键节点。

插件的接口也很简洁——只需要一个 name 字符串和一个 apply 函数。apply 函数接收 HookManager 实例，在里面调 tap 注册钩子就行。

插件加载支持三种来源：内置插件（比如 git-log、version-bump）、npm 第三方包、本地文件路径。plugin-loader 会根据 use 字段判断来源，动态 import 后调用 apply。

这套设计的好处是核心代码和业务逻辑完全解耦。核心流程只管"在什么时候调什么钩子"，至于钩子里做什么事情，全部交给插件。这就是开闭原则的体现——对扩展开放，对修改关闭。

---

## 六、平台策略层——策略模式 + 工厂模式的经典搭配

平台策略层是策略模式的典型应用。我定义了一个 BaseUploader 抽象基类，规定了四个抽象方法：upload()、preview()、validateConfig()、platformName。每个平台继承这个基类，实现自己的上传逻辑。

比如微信平台的 WechatUploader，它在构造函数里用 Zod Schema 校验配置，在 upload 方法里调用 miniprogram-ci SDK 执行上传。支付宝用的是 minidev SDK，百度用的是 swan-toolkit CLI，字节用的是 tt-ide-cli。每个平台的鉴权方式、SDK 调用方式都不一样，但对外暴露的接口是统一的。

然后我用一个 PlatformFactory 做工厂，内部维护一个注册表——一个平台名到构造函数的 Map。Runner 拿到平台名后，调 `PlatformFactory.create('wechat', config)` 就能拿到对应的上传器实例，完全不需要关心具体是哪个类。

工厂还提供了一个 register() 方法，允许外部注册自定义平台。比如将来要支持快手小程序，用户可以自己写一个 KuaishouUploader，调 register 注册进来，主流程一行都不用改。

---

## 七、配置体系——三级优先级合并

配置读取用的是 Cosmiconfig，它能自动在项目目录中查找 flashmini.config.ts、flashmini.config.js、.flashminirc.json、package.json 的 flashmini 字段等 6 种格式的配置文件，找到第一个就停。

我还做了三级配置合并：CLI 参数的优先级最高，其次是环境变量，最后是配置文件。这样在 CI 环境里，敏感信息比如密钥、Webhook URL 可以通过环境变量注入，不用写在配置文件里提交到 Git。

配置校验全量用 Zod Schema，而且所有的 TypeScript 类型都是从 Schema 通过 z.infer 自动推导出来的——我叫它"Schema 即类型"。这意味着我只维护一份 Schema 定义，运行时校验规则和静态类型自动保持一致，不存在两边不同步的问题。

另外我还提供了一个 defineConfig() 辅助函数。它本身什么都不做，只是原样返回配置对象，但通过泛型约束，用户在写配置文件的时候能享受完整的 IDE 类型提示和自动补全。这个技巧在 Vite、Nuxt 等项目中也很常见。

## 八、通知系统

通知系统支持三个渠道：飞书富文本卡片、钉钉 ActionCard、通用 Webhook。

飞书通知我构建了一个 Interactive Card 格式的消息，包含标题（带颜色主题——全部成功是绿色，有失败是橙色）、分支和 commit 信息、各平台状态、变更日志、@相关人员。

钉钉通知实现了 HmacSHA256 加签验证——钉钉的安全机制要求在 Webhook URL 上附加 timestamp 和 sign 参数，sign 是用密钥对 timestamp 做 HMAC-SHA256 签名再 Base64 编码。

通用 Webhook 支持模板变量替换，用户可以在 body 里写 `{{version}}`、`{{platforms}}` 这些占位符，发送时自动替换为实际值。

通知系统跟平台策略层一样，也是抽象基类 + 工厂的结构。而且通知内容通过 beforeNotify 瀑布流钩子暴露给插件，插件可以动态修改消息内容。

---

## 九、错误处理和重试

我定义了一个 FlashminiError 自定义错误类，携带 code（枚举错误码）和 platform（关联平台）两个结构化字段。错误码覆盖了 CONFIG_NOT_FOUND、CONFIG_INVALID、UPLOAD_FAILED、NETWORK_ERROR、AUTH_ERROR 等场景。CLI 层可以根据错误码给出针对性的提示。

上传操作内置了指数退避重试机制。默认重试 2 次，间隔按 delay * (i+1) 递增——第一次等 3 秒，第二次等 6 秒。这在 CI 环境的网络波动场景下非常有用。

另外上传支持 parallel（并行）和 serial（串行）两种模式，以及 continueOnError 选项——某个平台失败后是否继续上传其他平台。这些都是可配置的。

---

## 十、打包和发布

用 tsup 做打包，它底层基于 esbuild。我配置了双入口（index + cli）、双格式（ESM + CJS）、自动生成 .d.ts 类型声明。package.json 里通过 exports 字段分别指向 ESM 和 CJS 的产物，确保 import 和 require 都能正常工作。

bin 字段指向 bin/flashmini.js，这个文件只有一行 `import('../dist/cli.js')`，纯粹做引导加载。

---

## 十一、成果

这个工具覆盖了团队所有小程序项目的日常发布和 CI 流水线。单次发布从手动操作 4 个平台大约 20 分钟，缩短到一条命令 30 秒内完成。按我们团队的发版频率估算，每年大概能节省 300 小时的人工操作时间。

---

## 十二、总结——用到的设计模式

最后总结一下，这个项目一共用到了 6 种设计模式：

| 模式 | 用在哪 | 解决什么问题 |
|------|--------|-------------|
| 策略模式 | 各平台上传器 | 每个平台是独立策略，新增平台不改主流程 |
| 工厂模式 | PlatformFactory / NotifierFactory | 按名称创建实例，调用方不关心具体类 |
| 观察者模式 | HookManager | 生命周期钩子，解耦核心逻辑与扩展逻辑 |
| 模板方法模式 | BaseUploader / BaseNotifier | 定义流程骨架，子类实现具体步骤 |
| 上下文对象模式 | Context | 贯穿全流程传递状态，替代全局变量 |
| 插件模式 | 整个 plugins 目录 | 业务逻辑与核心逻辑隔离，开闭原则 |

这个项目虽然是一个工具类项目，但它的架构设计、设计模式运用、类型体系设计，我觉得是比较完整的，也是我在工程化方面比较有代表性的一个作品。

### 5. 六大钩子在流程中的触发时序

```
时间轴（从上到下）             钩子名              谁触发            典型插件用途
─────────────────────────────────────────────────────────────────────────
                               │
  所有平台上传前               │
  ─────────────────  →   beforeAll(ctx)      Runner.run()    git-log 填充 changelog
                               │                              修改版本号/描述
                               │
  ┌───── 对每个平台循环 ────────┤
  │                            │
  │  单平台上传前              │
  │  ───────────── →   beforeUpload          uploadOne()     打印"开始上传 xxx"
  │                    (ctx, platform)                        记录开始时间
  │                            │
  │  执行上传（含重试）         │
  │  retry(upload, 2, 3000)    │
  │                            │
  │  单平台上传后              │
  │  ───────────── →   afterUpload           uploadOne()     记录上传日志
  │                    (ctx, result)                          写入历史文件
  │                            │
  │  如果上传失败              │
  │  ───────────── →   onError               uploadOne()     报警通知
  │                    (ctx, error, platform)                 错误分析
  │                            │
  └────────────────────────────┤
                               │
  所有平台上传后               │
  ─────────────────  →   afterAll(ctx)       Runner.run()    version-bump 递增版本
                               │                              生成汇总报告
                               │
  通知发送前                   │
  ─────────────────  →   beforeNotify        各 Notifier     修改通知内容
                         (message, ctx)      .send()         加自定义字段
                         【瀑布流】                            改 @的人
                               │
─────────────────────────────────────────────────────────────────────────
```
